# Сценарии тестирования — система контроля аренды

Документ для приёмки. ФТ — функциональные тесты (что система делает), НФТ —
нефункциональные тесты (как система это делает: надёжность, безопасность,
производительность, удобство).

Автотесты на ключевые ФТ лежат в `backend/tests/` и запускаются `pytest`
(см. README). Сценарии ниже покрывают и автоматизированную, и ручную приёмку.

Легенда статусов оплаты периода: **pending** — ожидает, **paid** — оплачено,
**overdue** — просрочено.

## 1. Функциональные тесты (ФТ)

### ФТ-1. Управление объектами
| № | Шаг | Ожидаемый результат |
|---|-----|---------------------|
| 1.1 | Создать объект (название, адрес, статус, email владельца) | Объект появился в списке на главной |
| 1.2 | Открыть карточку объекта | Поля совпадают с введёнными |
| 1.3 | Изменить статус (свободен / выложен в YKT / занят) | Статус и бейдж обновились |
| 1.4 | Указать невалидный email владельца | Ошибка валидации (422), объект не создан |
| 1.5 | Удалить объект | Объект исчез из списка, открытие даёт 404 |

### ФТ-2. Заведение аренды и автоматические начисления
| № | Шаг | Ожидаемый результат |
|---|-----|---------------------|
| 2.1 | Занять объект вручную (ФИО, email, даты, сумма, день оплаты) | Аренда создана, статус объекта → «Занят» |
| 2.2 | После создания аренды | По месяцам срока созданы начисления с суммой = сумме аренды, статус pending |
| 2.3 | День оплаты = 31, в месяце 30/28 дней | Срок оплаты = последний день месяца (29 фев в високосный) |
| 2.4 | Дата окончания раньше даты начала | Ошибка валидации (422) |
| 2.5 | Загрузить `.txt` (ФИО, email, даты, сумма, день оплаты) | Поля разобраны, аренда создана (см. `examples/lease-sample.txt`) |
| 2.6 | Загрузить `.txt` без суммы/дня оплаты | Ошибка 400 с перечнем недостающих полей |

### ФТ-3. Напоминание арендатору за 3 дня
| № | Шаг | Ожидаемый результат |
|---|-----|---------------------|
| 3.1 | Срок оплаты периода наступает через 3 дня | Арендатору уходит письмо с суммой, периодом и ссылкой подтверждения |
| 3.2 | Повторный прогон планировщика в тот же день | Дубль письма не отправляется (один раз на период) |
| 3.3 | Срок оплаты дальше чем через 3 дня | Письмо не отправляется |

### ФТ-4. Подтверждение оплаты арендатором
| № | Шаг | Ожидаемый результат |
|---|-----|---------------------|
| 4.1 | Открыть ссылку подтверждения | Показаны объект, адрес, период, сумма, срок |
| 4.2 | Отправить корректный текст («оплатил аренду…») | Период → paid, владельцу уходит письмо «Оплата подтверждена» |
| 4.3 | Отправить текст без признаков оплаты («привет») | Ошибка 400, статус не меняется |
| 4.4 | Повторное подтверждение уже оплаченного периода | Ошибка 400 |
| 4.5 | Открыть несуществующий токен | Ошибка 404 «Ссылка недействительна» |

### ФТ-5. Учёт фактических оплат (частичные/долги)
| № | Шаг | Ожидаемый результат |
|---|-----|---------------------|
| 5.1 | Внести частичную оплату (меньше суммы) | amount_paid растёт, статус остаётся pending, владельцу не пишем |
| 5.2 | Доплатить до полной суммы | Статус → paid, владельцу уходит письмо об оплате |
| 5.3 | В таблице начислений видно начислено/оплачено/остаток | Цифры корректны по каждому месяцу |

### ФТ-6. Просрочка
| № | Шаг | Ожидаемый результат |
|---|-----|---------------------|
| 6.1 | Срок оплаты прошёл, период не оплачен | Владельцу уходит письмо «Нет оплаты», период → overdue |
| 6.2 | Повторный прогон | Дубль письма не отправляется |
| 6.3 | Период уже оплачен | Письмо о просрочке не отправляется |

### ФТ-7. Расторжение договора
| № | Шаг | Ожидаемый результат |
|---|-----|---------------------|
| 7.1 | Отметить договор расторгнутым | Новые начисления после даты расторжения не создаются, объект → «Свободен» |

### ФТ-8. Тестовые ручки писем (для проверки SMTP)
| № | Шаг | Ожидаемый результат |
|---|-----|---------------------|
| 8.1 | «Напоминание об оплате (тест)» | Письмо арендатору с теми же шаблонами, что в проде |
| 8.2 | «Оплата подтверждена (тест)» | Письмо владельцу |
| 8.3 | «Нет оплаты (тест)» | Письмо владельцу о неоплате |

### ФТ-9. Демо-данные
| № | Шаг | Ожидаемый результат |
|---|-----|---------------------|
| 9.1 | Первый запуск на пустой БД | Засеяны объекты заказчика (Кирова 1/2/3, Пояркова 6п/7п, Б.М., Грозовая, Орджоникидзе 44) |
| 9.2 | Перезапуск при заполненной БД | Повторного дублирования объектов нет |

### ФТ-10. Вход по логину/паролю
| № | Шаг | Ожидаемый результат |
|---|-----|---------------------|
| 10.1 | Открыть админку без входа | Редирект на `/login`; запросы к API без токена → 401 |
| 10.2 | Войти с неверным паролем | Ошибка «Неверный логин или пароль» (401), токен не выдан |
| 10.3 | Войти с верными логином/паролем | Выдан токен, открывается список объектов |
| 10.4 | Запрос к защищённому API с валидным токеном | 200, данные возвращаются |
| 10.5 | Открыть `/api/confirm/{token}` без входа | Доступно (публичная ссылка для арендатора), 401 не возникает |
| 10.6 | Нажать «Выйти» | Токен удалён, возврат на экран входа |

## 2. Нефункциональные тесты (НФТ)

### НФТ-1. Надёжность и устойчивость
- Планировщик не падает при ошибке отдельной задачи: исключение логируется, сервис продолжает работу.
- Все письма-уведомления идемпотентны (флаги `reminder_3d_sent_at`, `overdue_notice_sent_at`): повторный прогон не шлёт дубли.
- При недоступности SMTP отправка письма возвращает понятную ошибку (502 на тест-ручках), данные в БД не теряются.
- Планировщик дублирует ежедневный прогон интервальным фолбэком каждые 6 часов — пропуск 09:00 не критичен.

### НФТ-2. Безопасность
- Админка закрыта логином/паролем; пароль и секрет — только в `.env`. Пароль не
  хранится в БД, не возвращается клиенту; сравнение через `hmac.compare_digest`.
- Токен сессии — stateless HMAC-SHA256(секрет, логин), переживает рестарт, на сервере не хранится.
- Секреты (SMTP-пароль, AUTH_*, ключи) только в `.env`; файл в `.gitignore`, в репозиторий не попадает.
- Токен подтверждения оплаты — случайный, 32 байта (`secrets.token_urlsafe`), уникален на каждый период; перебор непрактичен.
- Валидация входных данных на границе (Pydantic): email, диапазоны дат, день оплаты 1..31, сумма > 0.
- CORS ограничен адресом фронтенда из настроек.
- Подтверждение оплаты доступно только по валидному токену; чужие периоды по токену не видны.

### НФТ-3. Производительность
- Списки объектов и начислений отдаются за < 300 мс на типовом портфеле (десятки объектов, сотни начислений).
- Задачи планировщика выбирают только релевантные периоды индексируемыми фильтрами (по дате/статусу), без выгрузки всей таблицы в память.

### НФТ-4. Удобство и локализация
- Весь интерфейс и письма — на русском; суммы в рублях с разделением разрядов (`35 000 ₽`), месяцы прописью.
- Понятные сообщения об ошибках (что заполнить, какие поля не разобраны из `.txt`).
- Адаптивная вёрстка (карточки/таблицы корректны на мобильном и десктопе).

### НФТ-5. Развёртывание и эксплуатация
- Полный запуск одной командой `docker compose up --build`.
- Конфигурация только через переменные окружения; смена SMTP/портов без правки кода.
- Схема БД создаётся автоматически при старте; данные переживают перезапуск (volume `pgdata`).
- Ручной прогон проверок доступен через `POST /api/jobs/run-now` (для демонстрации без ожидания 09:00).
- Health-check `GET /api/health` для мониторинга.

### НФТ-6. Сопровождаемость
- Бэкенд покрыт автотестами (`pytest`) на ключевые сценарии; прогон < 1 c на SQLite без внешних зависимостей.
- Бизнес-логика начислений вынесена в отдельный модуль (`app/periods.py`), почта — в `app/mail.py`.

## 3. Как прогнать автотесты ФТ
```bash
cd backend
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
pytest -q
```
Ожидаемо: все 22 теста зелёные. Покрывают ФТ-1…ФТ-6, ФТ-8 и ФТ-10 (вход/токены).
SMTP замокан — реальные письма в тестах не уходят.
